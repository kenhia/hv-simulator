# Sprint 005 — HTTP API + persistence + Docker

Implements milestone **M4** from `planning/004-project-plan.md`. This is the
sprint that turns the library into the **microservice** from the project's
original prompt: POST a ship and a flight plan, then GET where it is right now.

## Goal

Wrap the existing domain logic (`ephemeris`, `kinematics`, `flightplan`,
`clock`) in a FastAPI service backed by SQLite, so ships and flight plans are
durable and queryable over HTTP in real time. Filing a plan compiles it once and
**stores the absolute-time segments**; state queries are a DB lookup plus one
analytic kinematics evaluation — no recompute, no simulation loop. Ship it in a
Docker container.

## Decisions (from the Sprint 005 scoping questions)

- **Persistence**: store the filed plan *and* its compiled segments as rows.
  State queries read segments; recompilation happens only when a plan is filed.
- **Dashboards**: polling only (`GET …/state` on an interval). SSE/WebSocket
  push is deferred.
- **Docker**: Dockerfile + docker-compose land in this sprint.
- **`PUT /clock`**: gated behind an env flag (e.g. `HVSIM_DEV_CLOCK=1`), off by
  default, so production cannot fast-forward or change rate by accident.

## Scope

- **Persistence layer** (SQLAlchemy + SQLite): tables for ships, flight plans,
  waypoints, and compiled segments. A serialization boundary maps a
  `Segment`/`Trajectory` to rows and back (store the profile + origin/direction
  floats; rebuild the `Trajectory` on read).
- **FastAPI app** with the endpoint set from the plan (below). Pydantic v2
  request/response models.
- **Unit conversion at the boundary**: SI internally; responses give position in
  km **and** AU, velocity in km/s **and** fraction of c, acceleration in g, plus
  ETA, distance remaining, and percent-complete. No SI leakage in responses.
- **App-level `SimClock`** singleton as the one time source; `GET /clock` always,
  `PUT /clock` (set rate / jump) only when the dev flag is on.
- **Dockerfile + docker-compose** running the service over a mounted SQLite file;
  a documented `curl` sequence reproduces the canonical merchant run.

### Endpoints (from `planning/004`)

```
POST   /ships                      create a ship
GET    /ships                      list ships + summary state
GET    /ships/{id}                 ship + current state
GET    /ships/{id}/state?at=<ts>   full state vector (default: now)
POST   /ships/{id}/flightplan      file & compile a plan (stores segments)
GET    /ships/{id}/flightplan      active plan: segment timeline + ETAs
DELETE /ships/{id}/flightplan      abort the active plan (status → aborted)
GET    /bodies                     all bodies + current positions
GET    /bodies/{id}/state?at=<ts>  single body position
GET    /clock                      sim time, rate, epoch
PUT    /clock                      dev/test only — set rate or jump (env-gated)
GET    /health                     liveness (for Docker/compose)
```

## Out of scope

- **SSE/WebSocket push** — polling only this sprint.
- **`subjective_time_delta`** — the relativistic field is *reserved* in the
  state schema (returns null) but not computed yet; full calc is a later refinement.
- **Abort semantics beyond "mark aborted"** — `DELETE` sets the plan's status to
  aborted and the ship goes idle at the body it's nearest/heading-to per the last
  evaluable state; halt-in-place vs. auto-replan (plan open question) is deferred.
- **Hyperspace / wormhole segment kinds** — Phase 2.
- **Auth, multi-user, migrations framework** — single-process SQLite is fine for
  v1; schema is created on startup (Alembic later if needed).
- **Scheduling/replanning, multi-ship fleet ops** — later.

## Acceptance criteria

- **Canonical run over HTTP**: `POST /ships` (250 g / 0.6c merchant), then
  `POST …/flightplan` for Earth → Titan Station (6 h layover) → Earth; the
  returned timeline shows transit / layover / transit with the layover at
  exactly 6 h and a ~33 h overall ETA, matching the library-level numbers.
- **State across phases via HTTP**: `GET …/state?at=<ts>` returns the right
  phase and a human-unit position (km + AU) for predeparture, mid-transit
  (speed > 0, reported in km/s and fraction of c), layover, and arrived.
- **Persistence is real**: after recreating the app/session against the same
  SQLite file (simulating a restart), the ship, plan, and segments are intact
  and state is queryable **without** recompilation.
- **Bodies**: `GET /bodies` lists planets, moons, and stations with current
  positions; `GET /bodies/{id}/state?at=` resolves Titan Station.
- **Clock gating**: `GET /clock` always works; `PUT /clock` returns 403/404 when
  `HVSIM_DEV_CLOCK` is unset, and when set, jumping the clock changes the
  `state` reported by a no-`at` query.
- **No SI leakage**: every position/velocity/accel field in a response carries
  human units; raw metres/m·s⁻¹ never appear in the API contract.
- **Docker**: the image builds and `docker compose up` serves `GET /health`;
  the documented `curl` sequence reproduces the canonical run against the
  container. (If Docker is unavailable in the dev environment, note it and
  verify the image build + a local uvicorn run instead.)
- **Gate stays green**: `uv run pytest` (incl. API tests), `ruff check`,
  `ruff format --check` all clean.

## Notes / decisions

- Milestone mapping: this completes Phase 1. Phase 2 (universe builder,
  hyperspace, wormholes) adds new *segment kinds* and likely the move to
  Postgres — the segment table is shaped to absorb new kinds without a rewrite.
- IDs: string UUIDs for ships/plans (stable, opaque in URLs).
- The compiler and domain code stay pure and DB-free; persistence/HTTP is a thin
  shell around them. Keep `ephemeris`/`kinematics` import-clean of FastAPI/SQLA.
- `at` query param: ISO-8601 UTC; default is `SimClock.now()`.
- Schema created on startup via `create_all`; a real migration tool waits until
  the schema churns.
